# 문서 동기화 체커

코드 변경 사항을 분석하여, 영향 받는 문서만 선별적으로 점검하고 갱신한다.

## Step 1: 변경 사항 파악

인자가 주어지면 해당 ref 기준, 없으면 아래 순서로 변경 사항을 수집한다:

1. `git diff --staged --stat` + `git diff --staged` — staged 변경
2. `git diff --stat` + `git diff` — unstaged 변경
3. `git status --porcelain` — untracked 파일(?? 항목)은 내용도 Read로 수집
4. 1~3 모두 비어있으면 `git diff HEAD~1 --stat` + `git diff HEAD~1`

인자가 있을 경우: `git diff $ARGUMENTS --stat` + `git diff $ARGUMENTS`

**수집 결과물**: 변경된 파일 경로 목록 + diff 내용 전문.

## Step 2: 변경 영향도 분류

변경된 파일들을 아래 기준으로 분류한다. 하나의 변경이 여러 카테고리에 해당할 수 있다.

| 변경 카테고리 | 해당 조건 (예시) | 영향 받는 문서 Tier |
|---|---|---|
| **구조 변경** | 파일/디렉토리 추가·삭제·이동, 진입점 변경 | Tier 1, 2, 3 |
| **의존성 변경** | requirements.txt, package.json, go.mod, Cargo.toml, build.gradle 등 | Tier 1, 2 |
| **설정 변경** | .env 스키마, config 파일, secrets 구조, CI/CD 파일 | Tier 1, 2 |
| **API/인터페이스 변경** | 공개 함수 시그니처, REST 엔드포인트, CLI 인자 변경 | Tier 2, 3 |
| **비즈니스 로직 변경** | 핵심 알고리즘, 데이터 모델, 처리 흐름 변경 | Tier 1, 3, 4 |
| **기능 추가/삭제** | 새로운 기능 모듈 추가 또는 기존 기능 제거 | Tier 1, 2, 3, 4 |
| **비기능 변경** | 주석, 포맷팅, 리팩토링(동작 동일), 테스트만 변경 | 없음 (조기 종료) |

**조기 종료 조건**: 모든 변경이 "비기능 변경"에만 해당하면 Step 5로 바로 이동하여 "문서 영향 없음" 보고.

## Step 3: 문서 탐색 및 수집

영향 받는 Tier의 문서만 선별적으로 탐색한다.

### Tier 1: 에이전트 컨텍스트 (CLAUDE.md)
- 탐색: 프로젝트 루트의 `CLAUDE.md`, 하위 디렉토리의 `CLAUDE.md`
- 점검 항목:
  - 기술 스택 목록이 현재 의존성과 일치하는가
  - 프로젝트 구조 트리가 실제 파일 시스템과 일치하는가
  - 개발 규칙/컨벤션이 현재 설정과 일치하는가
  - 핵심 로직 설명이 코드 동작과 일치하는가

### Tier 2: 개발자 문서 (README 등)
- 탐색: 프로젝트 루트의 `README.md`, `CONTRIBUTING.md`, `CHANGELOG.md`
- 점검 항목:
  - 설치/실행 가이드가 현재 의존성·설정과 일치하는가
  - 사용법 예시가 현재 CLI/API와 일치하는가
  - 프로젝트 설명이 현재 기능 범위를 반영하는가

### Tier 3: 스펙/설계 문서
- 탐색: `docs/` 디렉토리 내 `.md` 파일 (PRD, TRD, ADR 등)
- **제외**: `node_modules/`, `vendor/`, `venv/`, `.venv/`, `build/`, `dist/`, `.git/` 하위
- 점검 항목:
  - 데이터 모델/스키마 설명이 코드와 일치하는가
  - 기능 명세가 구현된 동작과 일치하는가
  - 기술 아키텍처 설명이 현재 구조와 일치하는가

### Tier 4: 메모리 파일
- 탐색: `pwd`로 프로젝트 경로를 확인한 후, 해당 경로를 기반으로 Claude 메모리 디렉토리를 유추한다.
  - 경로 패턴: `~/.claude/projects/<path-with-dashes>/memory/`
  - 경로 변환 규칙: 프로젝트 절대 경로의 `:` 제거, `/` 또는 `\`를 `-`로 변환
  - 예: `D:\workspace\MyProject` → `~/.claude/projects/D--workspace-MyProject/memory/`
- 해당 디렉토리가 존재하면 내부 `.md` 파일을 모두 읽는다.
- 점검 항목:
  - 구현 상태(implementation status)가 실제 코드 상태와 일치하는가
  - 프로젝트 스펙 요약이 현재 PRD/TRD와 일치하는가
  - 기술적 주의사항이 여전히 유효한가

## Step 4: 문서 갱신

각 Tier별로 점검 결과를 처리한다:

### 갱신 원칙
1. **사실만 수정**: diff에서 확인된 변경 사항만 반영. 추측으로 내용을 추가하지 않는다.
2. **구조 유지**: 기존 문서의 형식, 어조, 언어를 그대로 유지한다.
3. **최소 변경**: 불일치 부분만 정확히 수정. 불필요한 리포맷팅을 하지 않는다.
4. **프로젝트 구조 트리**: 파일 추가/삭제/이동이 있을 때만 갱신한다. 트리를 갱신할 때는 `ls` 또는 Glob으로 실제 파일 시스템을 확인하여 정확하게 반영한다.

### Tier별 갱신 판단

| Tier | 갱신 기준 | 갱신 방법 |
|---|---|---|
| 1 (CLAUDE.md) | 구조·의존성·설정·로직 변경 시 | 직접 Edit으로 수정 |
| 2 (README 등) | 설치법·사용법·프로젝트 설명 불일치 시 | 직접 Edit으로 수정 |
| 3 (스펙 문서) | 스키마·기능 명세·아키텍처 불일치 시 | 직접 Edit으로 수정 |
| 4 (메모리) | 구현 상태·스펙 요약 불일치 시 | 직접 Edit으로 수정 |

## Step 5: 결과 보고

### 출력 형식

```
### 문서 동기화 결과

**변경 요약**: (변경 카테고리 나열, 예: 구조 변경, 의존성 변경)

| Tier | 문서 | 상태 | 변경 내용 |
|---|---|---|---|
| 1 | CLAUDE.md | 최신 / 갱신함 | (구체적 변경 설명) |
| 2 | README.md | 최신 / 갱신함 / 해당없음 | (구체적 변경 설명) |
| 3 | docs/TRD.md | 최신 / 갱신함 / 해당없음 | (구체적 변경 설명) |
| 4 | memory/status.md | 최신 / 갱신함 / 해당없음 | (구체적 변경 설명) |
```

상태 범례:
- 최신: 점검 완료, 수정 불필요
- 갱신함: 불일치 발견하여 수정 완료
- 해당없음: 이번 변경과 무관하여 점검 생략
- 수동 확인 필요: 자동 갱신이 어려운 경우 (대규모 구조 변경 등)

비기능 변경만 감지된 경우:
```
### 문서 동기화 결과

변경 내용이 문서에 영향을 주지 않습니다 (비기능 변경: 포맷팅/리팩토링/테스트).
```
